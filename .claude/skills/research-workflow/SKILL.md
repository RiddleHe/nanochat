# 0. Example research: Bank of Values (BoV)

The BoV paper (`papers/bank_of_values.pdf`) asked one focused question: in the deep transformer layers, what does the standard value projection (the matrix called `c_v` that turns the residual stream into the V used by attention) actually contribute? We tried several drop-in replacements for V — pulling V from layer 0, from layer 0's input, or just rescaling the existing V — and noticed they only behaved interestingly in the last third of layers. The paper's answer is that V in those late layers can be replaced by a per-layer lookup table indexed by the token id, which removes a matrix multiply and lets us avoid storing V in the KV cache at all. We reached that conclusion by running every variant in §2/§3/§4 through the same three-step pipeline below: build the variant, train it at a fixed compute budget so the comparison is fair, score it, and plot the results.

# 1. Model ablation pipeline

A new variant is born in three files. (1) In `nanochat/model/gpt_base.py`, add a boolean flag on the config dataclass `GPTBaseConfig` so you can turn the variant on; add a branch in `Attention.forward` that runs only for "target" layers (the last third by default — the helper `_is_target_layer` does this); if your variant has new learnable numbers like a scalar coefficient, declare them in `__init__` and set their starting values in `init_weights`. (2) In `nanochat/model_registry.py`, add a small config class that flips your flag on, and register a name in the model dictionary. (3) Train it.

Training always runs from the repo root, after exporting `NANOCHAT_BASE_DIR=/local-ssd/mh3897` so everything lands in the right place, and you call Python through `uv` (e.g. `uv run torchrun ...`) because the project's environment is managed by uv — a plain `python` may pick up the wrong one. The easiest launcher is `runs/ablations.sh <your_model_name>`. The script knows the compute-optimal FLOPs target for each depth (`1.5e18` for d12, `3.91e19` for d24, both ~10.5 tokens per parameter), auto-picks the batch size from the Power Lines scaling law, and writes into a depth-specific folder (`arch_d12_*` or `arch_d24_*`). That folder is what makes a lineage: the script skips anything that already has a checkpoint there, so new variants stay directly comparable to old ones. A useful trick when running a new variant is to pass *every* existing tag as a positional argument too — they'll all auto-skip the train step but get re-extracted into `val_loss_curves.csv` for a single comparison file. The number of GPUs is otherwise free: you can run on 4 H100s normally, or on just 2 if the main bank is busy, and the loss curves still align on the FLOPs axis because total batch size (in tokens) is held constant — only the wall-clock per step changes. Subtle bits: don't change `--target-flops`, batch size, or `TAG_PREFIX` partway through a lineage (the learning-rate schedule depends on the token budget and changing it mid-stream breaks the comparison); if a multi-day run dies, resume with `--resume-from-step <N>`, but only with the same number of GPUs since each rank saves its own optimizer shard.

When adding a new config flag, follow these rules so old checkpoints still load: (1) add the new field to `GPTBaseConfig` with a default (usually `False`) — dataclasses tolerate extra defaulted fields when reading old JSON; (2) gate any new `nn.Parameter` creation on the flag, otherwise old state dicts fail `strict=True` loading; (3) in `init_weights`, guard the new inits with `if hasattr(...)`; (4) in `setup_optimizer`, route any new scalar into the right param group (alpha-style if it starts at 1.0, beta-style if it starts at 0.0) and add its size to the `coeff_numel` exclusion in `estimate_flops`; (5) never rename or remove existing config fields, since each train log's "Model config:" line names them by string and the CSV extraction grep depends on the format being stable. Quick verification before launching: load the latest baseline checkpoint with the new code path; if it loads `strict=True` and one forward step produces the same loss as before, the rest of the principled set is safe.

# 2. Model evaluation pipeline

Evaluation happens in two stages. While the model trains, every few hundred steps it computes the validation bits-per-byte ("val bpb") on a held-out slice of the ClimbMix dataset (about 42 million tokens) — that's our language-modeling signal and what the training curves in the paper are made of (the numbers are inside each `arch_d{12,24}_gpt_base_*_train.log`, on lines beginning with "Step ..." and "Validation bpb:"). The second stage runs *after* training: `scripts/base_eval.py` evaluates the model on the CORE benchmark suite — 21 tasks from DCLM CORE listed in `configs/core.yaml`, where 12 are multiple-choice (scored by picking the highest-log-probability answer) and 9 are short-answer generation (scored by exact match). Each task's random-guessing baseline is in `eval_bundle/eval_meta_data.csv`. Output CSVs land in `/local-ssd/mh3897/base_eval/` with three columns: task name, raw accuracy, and a "centered" score that subtracts the baseline.

When we write up results, the convention is: report the **raw accuracy** for each task in the paper table, and add one summary number — **CORE = the average of the 21 centered scores**. Centered scores make "no better than guessing" map to zero, so a reader can see at a glance whether a model is actually doing anything on each task. Keep noise in mind too: small datasets (e.g. COPA with 100 questions) have a per-task error bar of several percent, so don't chase tenths of a point. There's a second eval pipeline, `scripts/base_eval_extended.py`, which runs MMLU/BBH through the standard lm-eval-harness — useful for comparison with published numbers but mostly redundant for us, and a quirk to know is that those tasks use a deterministic "first-N" few-shot sampler, so averaging over seeds gives the same number every time.

# 3. Visualization style

All figures in the paper use the same minimal look: STIX serif (a Times look-alike that's free) for every label, only the top and bottom borders of each panel drawn (left and right hidden), and faint horizontal grid lines. A small fixed color palette carries meaning across plots: blue for the baseline or the V₁ source, peach for BoV or the x₀ source, teal for the standard V (or x_i), and green when we need a fourth distinct color (e.g. the qwen3 tokenizer curve or the SVFormer baseline). For paper-ready figures, set the matplotlib figure size to exactly 6.3 × 2.3 inches (one ACL column width) with titles 10 pt, labels and legends 9 pt, ticks 8 pt — and importantly, save *without* `bbox_inches='tight'`, otherwise LaTeX would scale the cropped image up to column width and shrink your font sizes. Output everything to `./results/` (gitignored), not under `/local-ssd/` — that path is reserved for run artifacts (checkpoints, train logs, CSVs). For plots that read learned coefficients from checkpoints, the working template is `scripts/inspect/visualize_value_coeffs.py`; two earlier scripts (`visualize_alpha_beta.py`, `visualize_gamma_v.py`) cover narrower cases and are mostly superseded by it.
